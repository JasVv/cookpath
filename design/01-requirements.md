# 01. 要件定義

## 1.1 アプリケーション名

**Cookpath -メニュー記録と買い物サポート-**

画面左上のヘッダーに常時表示する。

## 1.2 目的

個人利用者が以下を効率的に行うことを支援する。

1. **1週間程度の献立の計画立案**
2. **その献立に必要な食材を一覧化した買い物チェックリストの作成**

ネットスーパーでの実際の購入操作はユーザーが行う（本システムの責務外）。

## 1.3 利用者・利用環境

| 項目 | 内容 |
|---|---|
| 想定利用者 | 1名（個人専用） |
| 同時利用 | 想定しない |
| 端末 | PC（一般的なモニタサイズ）のみ |
| ブラウザ | モダンブラウザ（Chrome / Edge / Safari / Firefox の最新版） |
| ネットワーク | オフラインでも動作可能（フロントエンド完結のため） |

スマホ・タブレットでの最適化は行わない。

## 1.4 主要ユースケース

### UC-1: レシピを登録する
材料（材料名・分量・単位）と簡潔な手順メモを持つレシピを登録・編集・削除する。

### UC-2: 献立カレンダーに料理を登録する
日付を選び、その日の献立として複数の料理を登録する。料理は以下のいずれかの方法で指定する。
- 登録済みレシピから選択（材料・分量はその日だけ変更可能）
- 料理名と材料を直接入力（料理名のみ・材料なしも可）

### UC-3: 1週間分の献立を計画する
当日を起点とした1週間カレンダーに対して、月カレンダーや別の日からドラッグ&ドロップで献立をコピー／入れ替えして、計画を組み立てる。

### UC-4: 過去の献立を参考に新しい献立を立てる
過去数ヶ月分の月カレンダーを遡って閲覧する。

### UC-5: 買い物チェックリストを作る
期間（デフォルト: 当日から1週間）を指定し、その期間内の全料理の材料を集計したチェックリストを生成。チェックを付けながら買い物に使う。

### UC-6: データをバックアップ／復元する
JSON形式でエクスポート／インポートする。ブラウザのデータ消失リスクへの対策。

## 1.5 機能要件サマリ

| 機能 | 概要 |
|---|---|
| 献立カレンダー（月） | 月単位表示。当日強調。月送り。日付クリックで献立入力モーダル。 |
| 献立カレンダー（週） | 当日起点1週間。1日ずつ前後に動かせる。月カレンダーと相互ドラッグ&ドロップ可能。 |
| 献立入力モーダル | レシピ選択（検索ボックス付きリスト）／直接入力。料理ごとに材料編集・並び替え・削除。レシピ更新・新規保存ボタン。 |
| レシピ管理 | 一覧（検索・並び替え）、モーダルでの新規・編集、削除。 |
| 買い物チェックリスト | 期間指定。日付順／材料名順切替。出典表示。チェックは画面内一時状態。 |
| 設定 | JSON export/import（インポートは全置換）、Persistent Storage 要求、最終 export 日時表示・30日警告。 |

## 1.6 非機能要件

| 項目 | 内容 |
|---|---|
| データ保存 | ブラウザ内（IndexedDB）。Persistent Storage を要求して退避耐性を上げる。 |
| バックアップ | JSON export/import を提供。30日間 export していない場合に警告表示。 |
| インポート挙動 | 既存データを全置換する（マージ・選択モードは設けない）。 |
| パフォーマンス | 数年分の献立 + 数百件のレシピを想定。月カレンダー表示で体感遅延が出ないこと。 |
| 言語 | UI文言は日本語。コードの識別子（変数・関数・クラス名等）は英語。 |

## 1.7 スコープ外

- ネットスーパーAPI連携・カート投入
- 価格情報の取得・比較
- 栄養素計算・カロリー管理
- 在庫管理（冷蔵庫・パントリー）
- レシピ画像のアップロード（テキスト管理のみ）
- マルチユーザー・アカウント機能
- スマホ・タブレット最適化
- 朝/昼/夕の区分（1日1食扱い、内部に複数料理を持つ）
- 材料の表記ゆれ吸収・自動マージ・単位変換
- 印刷機能
